# Matchbook вЂ” Architecture

> Self-hosted agency finance ops. Automated invoice collection в†’ LLM reconciliation в†’ expense dashboard в†’ accountant reports.

---

## Overview

Matchbook is a self-hosted tool for freelancers and small agencies to automate invoice collection, reconcile expenses against bank statements using an LLM, and produce flexible reports for their accountant.

**Core flow:**
```
Gmail / Supplier Portals в†’ LLM Extraction в†’ Google Drive + Postgres в†’ LLM Matching в†’ Review UI в†’ Accountant Report
```

---

## Application Modules

| Module | Description |
|---|---|
| **Dashboard** | Monthly spend overview, expenses by category/vendor, VAT summary, MoM comparison, missing invoice alerts |
| **Collection** | On-demand Gmail sync, portal scraper triggers, extraction review |
| **Invoices** | Invoice list, detail view, Drive links, vendor history, category tagging |
| **Reconciliation** | Bank statement upload, LLM match review, exception flagging, period-based workflow |
| **Vendors** | Manage vendors, set billing cycle (monthly/bimonthly/quarterly/annual), view invoice history |
| **Reports** | PDF + Excel export, flexible timeframe selection (single month, quarter, year, custom range) |

## Pipelines

### Pipeline 01 вЂ” Invoice Collection

Triggered on-demand from the UI. No cron вЂ” user runs it when ready.

```
Gmail Inbox (invoices@domain в†’ free Gmail via Cloudflare routing)
    в†“
Gmail API вЂ” fetch all unread emails with PDF attachments (on-demand trigger)
    в†“
Playwright scrapers вЂ” per-supplier portal connectors (separate on-demand trigger per supplier)
    в†“
LLM Extraction (Claude API) вЂ” vendor, amount ex/inc VAT, VAT rate, invoice #, date, IBAN
    в†“
Structured rename вЂ” 2026-03_Vercel_в‚¬49.00_INV-1234.pdf
    в†“
Postgres вЂ” invoice metadata + job_run record
    в†“
Google Drive вЂ” /Invoices/2026/March/ (auto-organised)
```

### Pipeline 02 вЂ” Reconciliation

Triggered on-demand per period from the UI.

```
Bank statement upload вЂ” XLS / XLSX / CSV / MT940 / CAMT.053
    в†“
StatementIngester вЂ” converts file to clean text representation
    (XLS/XLSX в†’ openpyxl в†’ CSV string, MT940 в†’ mt940 lib в†’ text, CAMT.053 в†’ XML parse в†’ text, CSV в†’ decode)
    в†“
LLM Parsing (Claude API) вЂ” reads raw text, extracts normalised transaction list
    Returns: [{ tx_date, amount, description, counterparty, counterparty_iban }]
    в†“
Type casting + validation вЂ” enforce numeric(10,2), date types, required fields
    в†“
Postgres вЂ” transactions stored, period derived from date range
    в†“
LLM Matching (Claude API) вЂ” reasons across all invoices + transactions for the period
    Returns: match pairs with confidence score (0вЂ“1) + rationale text
    в†“
React review UI вЂ” confirm / reject / reassign matches, flag exceptions
    в†“
Postgres вЂ” confirmed matches written, invoice + transaction status updated
```

**Why LLM parsing:** Bank statement formats vary significantly across institutions and change over time. Using the LLM to parse the statement means format differences are handled automatically without brittle column-mapping code вЂ” the ingester only needs to get the file into readable text, not interpret its structure.

### Pipeline 03 вЂ” Reporting

Triggered on-demand from the Reports module. Supports flexible timeframes.

```
User selects timeframe + format
    в†“
Timeframe options:
    - Single month        e.g. March 2026
    - Quarter preset      Q1/Q2/Q3/Q4 + year  в†’  e.g. Q1 2026 = JanвЂ“Mar
    - Year to date        Jan в†’ current month
    - Full year           all 12 months of a given year
    - Custom range        from month в†’ to month  e.g. Nov 2025 в†’ Mar 2026
    в†“
Backend aggregates invoices + matches across all periods in range
    в†“
Report generated in chosen format:
    PDF   вЂ” WeasyPrint, paginated, accountant-ready
    Excel вЂ” openpyxl, multi-sheet, filterable
    в†“
Download served directly to browser
```

**Report contents (all timeframes):**
- Cover page: agency name, timeframe label, generated date
- Summary: total spend ex VAT, total VAT, invoice count, matched/unmatched вЂ” broken down per month if multi-period
- Expenses by category (totals)
- Expenses by vendor (totals)
- Full invoice list: vendor, invoice #, date, amount ex VAT, VAT, amount incl VAT, matched tx date, status
- Unmatched transactions (flagged)
- Missing invoices: vendors with a billing cycle that expected an invoice in range but none received

## Type Generation

Frontend TypeScript types and RTK Query hooks are generated from the FastAPI OpenAPI schema using **Orval**. The backend and frontend are two standalone projects; type generation is a one-shot script run from the frontend after backend routes change.

### Flow

```
Pydantic models + FastAPI response_model=
    в†“
backend/export_schema.py  в†’  openapi.json  (repo root, committed)
    в†“
frontend: pnpm generate:api  (runs Orval)
    в†“
frontend/src/api/           (generated, gitignored)
    в”њв”Ђв”Ђ types.ts             all request/response schemas
    в”њв”Ђв”Ђ invoices.ts          RTK Query hooks for /invoices
    в”њв”Ђв”Ђ reconciliation.ts    RTK Query hooks for /reconciliation
    в”њв”Ђв”Ђ vendors.ts
    в”њв”Ђв”Ђ dashboard.ts
    в”њв”Ђв”Ђ reports.ts
    в””в”Ђв”Ђ jobs.ts
```

### Schema export

`backend/export_schema.py` dumps the OpenAPI schema without starting the server:

```python
# backend/export_schema.py
import json
from app.main import app

with open("../openapi.json", "w") as f:
    json.dump(app.openapi(), f, indent=2)
```

Run with: `python export_schema.py`

### Frontend scripts

```json
// frontend/package.json
"scripts": {
  "generate:api": "orval --config orval.config.ts",
  "dev": "vite",
  "build": "tsc && vite build",
  "test": "vitest"
}
```

### Orval config

```ts
// frontend/orval.config.ts
import { defineConfig } from 'orval'

export default defineConfig({
  matchbook: {
    input: '../openapi.json',
    output: {
      mode: 'tags-split',         // one file per router tag
      target: 'src/api',
      client: 'react-query',      // RTK Query via redux-toolkit adapter
      schemas: 'src/api/types',
    },
  },
})
```

### FastAPI conventions required for clean output

Orval output quality depends on the backend being explicit. Follow these rules throughout:

- Every route must declare `response_model=` вЂ” never rely on inferred return types
- Every router must set `tags=["invoices"]` (or the relevant module name) вЂ” Orval splits files by tag
- Use Pydantic models for all request bodies; never use raw `dict`
- Enums must be `str` enums so OpenAPI serialises them as string literals
- Nullable fields must be typed as `Optional[X]`, not just `X | None` (for OpenAPI compat)

### When to regenerate

Re-run `python export_schema.py && pnpm generate:api` after:
- Adding or removing any route
- Changing a request or response Pydantic model
- Changing a route's `response_model`

`openapi.json` is committed to the repo root so the frontend can generate types without running the backend.

## LLM Usage

### Extraction prompt (Pipeline 01)
Input: raw PDF text extracted by pdfplumber
Output: structured JSON
```json
{
  "vendor": "Vercel Inc.",
  "invoice_number": "INV-2026-0312",
  "invoice_date": "2026-03-01",
  "amount_excl": 40.50,
  "amount_incl": 49.00,
  "vat_amount": 8.50,
  "vat_rate": 21.0,
  "currency": "EUR",
  "iban": "NL02ABNA0123456789"
}
```

### Statement parsing prompt (Pipeline 02, step 2)
Input: raw text representation of bank statement, pre-processed by StatementIngester
Output: structured JSON array
```json
[
  {
    "tx_date": "2026-03-04",
    "amount": -49.00,
    "description": "VERCEL INC SUBSCRIPTION",
    "counterparty": "Vercel Inc",
    "counterparty_iban": "NL02ABNA0123456789"
  }
]
```

Amount sign convention: debits are **negative**, credits are **positive**. Prompt handles all bank conventions:
- Signed amount column вЂ” use sign directly
- Separate debit/credit columns (common in Greek banks: Piraeus, Alpha, Eurobank) вЂ” debit в†’ negative, credit в†’ positive
- Unsigned amount + type column вЂ” infer from label

### Matching prompt (Pipeline 02, step 3)
Input: all unmatched invoices + all unmatched transactions for the period + known vendor aliases
Output: match suggestions
```json
[
  {
    "invoice_id": "uuid",
    "transaction_id": "uuid",
    "confidence": 0.97,
    "rationale": "Amount matches exactly (в‚¬49.00). Date within 2 days. 'VERCEL INC' in bank description matches vendor name."
  }
]
```

## Reporting вЂ” Timeframe Reference

| Option | Description | Resolves to |
|---|---|---|
| Single month | One period | e.g. `[2026-03]` |
| Quarter | Q1вЂ“Q4 of a year | Q1 2026 в†’ `[2026-01, 2026-02, 2026-03]` |
| Year to date | Jan в†’ current month | `[2026-01 вЂ¦ 2026-03]` if run in March |
| Full year | All 12 months | `[2026-01 вЂ¦ 2026-12]` |
| Custom range | User picks from/to month | e.g. `[2025-11 вЂ¦ 2026-03]` |

Multi-period reports include a per-month breakdown row in all summary tables. UI uses `@mantine/dates` `MonthPickerInput` for the custom range picker.
